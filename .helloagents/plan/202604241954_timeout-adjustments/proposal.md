## 1. 需求

### 背景
当前项目中 `LLM`、`Tikhub` 和生图接口的超时分散在租户默认值、工具默认值和调用点硬编码中，默认值偏短，导致长耗时模型调用和第三方接口更容易在正常执行中超时。

### 目标
- 将 `LLM` 运行时默认超时统一提升到 `600s`
- 将 `Tikhub` 相关接口默认和显式调用超时统一提升到 `300s`
- 将生图接口超时提升到 `600s`
- 同步更新测试和对外文档中的默认值描述

### 约束条件
```yaml
时间约束: 本次直接在现有实现上调整，不做额外配置中心重构
性能约束: 仅调整超时与相关文档，不改变重试次数与调用协议
兼容性约束: 保留显式传入 timeout_seconds 的覆盖能力
业务约束: 修改后常规租户运行需实际生效为 LLM 600s，而不是只修改兜底值
```

### 验收标准
- [ ] 租户默认 `timeout_seconds` 变为 `600`
- [ ] `LLM` 主调用默认读取到 `600s`
- [ ] `Tikhub` 主页解析、笔记抓取、分页抓取相关超时为 `300s`
- [ ] 生图接口超时为 `600s`
- [ ] 测试与文档中的默认值已同步

---

## 2. 方案

### 技术方案
直接调整现有默认值与显式调用点，不新增抽象层。具体包括：
- 提升租户 schema、数据库表默认值、tenant model 兜底值为 `600`
- 提升 `workflow/core/ai.py` 现有 `LLM` 默认路径实际使用到的租户超时
- 将 `Tikhub` 与主页解析相关工具函数默认值和节点调用值统一改为 `300`
- 将图片生成接口请求超时改为 `600`
- 更新测试桩与 API/README 示例，确保默认值一致

### 影响范围
```yaml
涉及模块:
  - app: 租户 API 默认超时字段
  - model: tenants 表默认值与运行时配置映射
  - workflow/core: LLM 超时来源
  - workflow/flow/content_create: Tikhub 与生图超时
  - workflow/flow/content_collect: Tikhub 调用超时
  - workflow/integrations: 热榜接口超时
  - tests: 默认值断言同步
  - docs: README 与 api.md 示例同步
预计变更文件: 12
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 只修改 LLM 兜底值但未修改租户默认值，导致常规路径仍为 30 秒 | 中 | 同步修改 schema、DB 默认值和 tenant 运行时兜底 |
| 测试仍使用旧默认值导致回归失败 | 低 | 同步更新默认值测试桩，仅保留显式自定义超时测试 |
| 文档与代码值不一致 | 低 | 同步更新 README 和 api.md 示例 |
