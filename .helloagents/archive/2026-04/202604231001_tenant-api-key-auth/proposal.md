## 1. 需求

### 背景
当前服务所有业务接口都可以直接访问，没有任何请求级鉴权能力。用户希望为每个用户/租户增加独立 API key，用于识别权限，且本次要求统一通过 `X-API-Key` 请求头校验。

### 目标
- 为每个租户增加独立 API key 字段
- 为除 `/health` 外的所有业务接口增加 `X-API-Key` 鉴权
- 保持现有统一响应包装和租户工作流能力不变

### 约束条件
```yaml
时间约束: 无
性能约束: 鉴权逻辑应在请求入口快速完成，不引入额外远程调用
兼容性约束: 现有 PostgreSQL schema 需兼容升级；已有租户需要可补齐 api_key
业务约束: 本次只支持 `X-API-Key`，不扩展到 JWT/OAuth
```

### 验收标准
- [ ] `tenants` 表具备 `api_key` 字段，并能通过接口读写返回
- [ ] 除 `/health` 外的业务接口都要求 `X-API-Key`，缺失或错误时返回统一错误响应
- [ ] 现有接口测试覆盖有效 key、缺失 key、错误 key 三类场景

---

## 2. 方案

### 技术方案
在 PostgreSQL `tenants` 表新增 `api_key` 列，并在自动建表/迁移逻辑中补齐。租户创建/更新接口支持写入和返回该字段。FastAPI 增加统一鉴权依赖，从 `X-API-Key` 读取请求头，并结合路径中的 `tenant_id` 或请求体里的 `tenant_id` 在数据库中校验。对于租户管理与流程运行接口，统一在入口依赖中完成校验；`/health` 保持免鉴权。

### 影响范围
```yaml
涉及模块:
  - app: 请求鉴权依赖、路由入口和响应模型更新
  - model: 租户 schema、读写逻辑和迁移逻辑更新
  - tests: 路由鉴权与回归测试补齐
  - knowledge-base: 记录新的接口鉴权规则
预计变更文件: 9
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 老租户缺少 api_key 导致接口全部不可用 | 中 | 提供自动兼容列迁移，并允许通过现有租户更新接口补齐 |
| 不同接口提取 tenant_id 位置不同导致漏鉴权 | 中 | 在统一依赖中同时覆盖 path 参数和请求体 `tenant_id` |
| 把 `/health` 也纳入鉴权影响探活 | 低 | 明确保留 `/health` 免鉴权 |

## 5. 技术决策

> 本方案涉及的技术决策，归档后成为决策的唯一完整记录

### tenant-api-key-auth#D001: 统一使用 X-API-Key 承载租户级 API key
**日期**: 2026-04-23
**状态**: ✅采纳
**背景**: 用户明确要求每个用户增加 API key 做权限鉴定，并指定采用 `X-API-Key` 头传递。
**选项分析**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A: `X-API-Key` | 实现简单，语义明确，适合单密钥服务鉴权 | 不是通用 OAuth 风格 |
| B: `Authorization: Bearer` | 更接近通用 token 风格 | 会和未来 JWT/OAuth 语义混淆 |
**决策**: 选择方案 A
**理由**: 当前需求是租户级静态密钥校验，不需要引入更复杂的 token 体系。
**影响**: 影响租户模型、接口依赖、请求测试和知识库中的接口规则说明
